# F-0122: Add YAML Frontmatter to All Agent and Skill Templates

**Type:** Enhancement
**Priority:** P1 (important)
**Status:** Approved
**Requested by:** PO
**Date:** 2026-03-26

## Problem

Claude Code now supports YAML frontmatter in agent files (`.claude/agents/*.md`) and skill files (`.claude/skills/*.md`). Frontmatter enables structured metadata — model selection, tool restrictions, permission modes, slash-command naming, and auto-invocation control — that Claude Code reads at load time to configure agent and skill behavior.

Currently, **none** of the 8 agent templates or 21 skill templates generated by appteam include frontmatter. This means generated teams miss out on:

- Explicit model pinning per agent (the wizard already collects model preference but only documents it in prose)
- Tool restrictions for non-coding agents (PM, TPM, Reviewer should not use Bash/Edit/Write)
- Permission mode configuration (e.g., `acceptEdits` for SWEs)
- Proper slash-command registration for skills (name, description, argument hints)
- Control over auto-invocation (`disable-model-invocation`) for workflow-heavy skills
- Skill-to-agent binding via the `agent` field

Adding frontmatter to all templates brings appteam-generated configurations in line with the Claude Code platform's current capabilities.

## Requirements

### Agent Frontmatter (8 templates)

1. All agent templates MUST begin with YAML frontmatter between `---` markers before any markdown content
2. Every agent MUST include the `name` field (lowercase-hyphen format matching the generated filename without `.md`)
3. Every agent MUST include the `description` field (one sentence explaining when Claude should delegate to this agent)
4. Agents SHOULD include the `model` field, using the `{{.ModelName}}` template variable where applicable (maps to the wizard's model selection)
5. Non-coding agents (PM, TPM, Reviewer) SHOULD include `disallowedTools` to restrict file-editing tools (`Bash`, `Edit`, `Write`) since these agents must never write application code directly
6. The SWE parametric template MUST use `{{.Number}}` in the `name` field to produce `swe-1`, `swe-2`, etc.
7. The custom agent template MUST use the template variable for the agent's name (lowercase-hyphen)
8. Agent frontmatter fields MUST be ordered: `name`, `description`, `model`, then any additional fields

### Skill Frontmatter (21 templates)

9. All skill templates MUST begin with YAML frontmatter between `---` markers before any markdown content
10. Every skill MUST include the `name` field matching the slash-command trigger (e.g., `spec`, `deploy`, `brainstorm`)
11. Every skill MUST include the `description` field (one sentence explaining what the skill does)
12. All skills SHOULD set `user-invocable: true` (they are all meant to be invoked via `/command`)
13. Workflow-oriented skills (brainstorm, cuj-list, cuj-test, roadmap, standup) SHOULD set `disable-model-invocation: true` to prevent auto-triggering, since these require interactive user participation
14. Skill frontmatter fields MUST be ordered: `name`, `description`, `user-invocable`, then any additional fields

### Template Integration

15. Frontmatter MUST be inserted as the very first content in each template string constant, before the existing `# Title` heading
16. The `---` markers and YAML content MUST be part of the Go string constant (not generated separately)
17. Existing template variables (`{{.ProjectName}}`, `{{.OwnerName}}`, etc.) MUST continue to work within frontmatter values
18. The existing markdown body content MUST remain unchanged (no modifications to headings, sections, or instructions)

### Test Updates

19. Existing tests that check template output MUST be updated to account for the new frontmatter prefix
20. Tests SHOULD verify that frontmatter begins with `---\n` and ends with `---\n` before the markdown body
21. Tests SHOULD verify that `name` and `description` fields are present in the rendered output

## Out of Scope

- Adding frontmatter fields beyond `name`, `description`, `model`, `disallowedTools`, `user-invocable`, and `disable-model-invocation` (e.g., `hooks`, `mcpServers`, `isolation`, `initialPrompt` can be added in a future enhancement)
- Wizard changes to let users customize frontmatter fields interactively
- Scion framework templates (scion_system_prompt.go, scion_agent_yaml.go, scion_agents_md.go) — these use a different format
- Non-agent/skill templates (claude_md.go, backlog.go, progress.go, releasenotes.go, pipeline.go, spec_template.go)

## Dependencies

- None — this is a self-contained enhancement to existing templates

## Implementation Notes

- Agent frontmatter uses `disallowedTools` (camelCase) per the Claude Code agent spec
- Skill frontmatter uses `disable-model-invocation` and `user-invocable` (kebab-case) per the Claude Code skill spec
- The PM, TPM, and Reviewer `disallowedTools` restriction enforces the existing "never write application code directly" rule at the platform level, not just in prose instructions
- Template variables in frontmatter values must be valid YAML when rendered — ensure descriptions don't contain YAML-breaking characters (colons, quotes). Use template pipeline functions if needed
- The custom agent template's `name` field should transform `{{.Name}}` to lowercase-hyphen format. Since the wizard already collects the name, consider whether the Go code or the template handles normalization
